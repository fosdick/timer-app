# Working with Jarvis

Explicit preferences, stated directly — these override Claude defaults. (Mirrors the live memory
file `working_style.md`; as of 2026-06-10.)

## 1. Full scope now, not ship-and-iterate

Jarvis spent his career in startups where a feature took a sprint, so "MVP it, refactor later"
was correct then. With Claude as the executor that calculus is broken — doing it right now costs
no more wall-clock than doing it partially. He will *always* pick "do all the work now" over a
staged approach. Don't reflexively carve scope ("defer X to v1.5") unless there's an actual
sequencing reason (e.g., a learning is needed for a downstream decision).

## 2. Don't slip into build mode during brainstorms

Signals he's brainstorming: "sound board", "brainstorm", "any ideas", "what do you think",
"give me your feedback". In that mode he wants: reflection of what he said, original thinking,
tensions named, questions that help *him* think. NOT: task lists, "should I start building?",
decision-form question blocks, premature code.

Signals to execute: "let's build", "make it so", "do it", "go", "let's give this a go".
Then: execute the full thoughtful scope. When in doubt, ask which mode — don't guess.

## 3. Division of labor

He owns planning and design; execution is on Claude — he won't be code-reviewing. He's
experienced enough to know the risks of commands without reading each one.

## 4. Practical details

- He tests on his own iPhone in real morning pranayama practice; iterations often land the
  night before. "I need a working app for the morning" is a real constraint.
- Audio quality judgments are his (by ear, in practice); measurements inform but don't decide.
- He values memory/docs hygiene — record decisions, keep them current.
